STEP 1 — Intent & Input Parsing Engine

Max-Logic • Truth-Only • Zero Assumptions

1. Position in the Global Reasoning Chain

STEP 1: Intent Parsing (What is truly requested?)
STEP 2: Context Parsing (What exists now?)
STEP 3: Cross-Reference (Is it structurally valid?)

STEP 1 is the logical gatekeeper. No system action, analysis, or planning is allowed before intent is normalized into machine-truth.

2. Core Truth

Human requests are not executable instructions.

They are subjective, incomplete, and often contradictory. STEP 1 exists to convert human language into an objective execution intent.

3. Raw Input Definition

Raw Inputs:
- User natural language request
- Conversation history
- Project state indicators
- User permissions & safety scope
- Execution environment metadata

Natural language is treated as untrusted input.

4. What "Intent" Means (Strict Definition)

Intent is the minimal, testable, non-emotional representation of a request. 

Intent {
  objective: string
  desired_outcome: string
  allowed_actions: string[]
  forbidden_actions: string[]
  constraints: {
    time?: number
    technology?: string[]
    safety_level: low | medium | high
  }
  assumptions: string[]
  ambiguity_level: low | medium | high
  execution_risk: low | medium | high
}

5. Internal Architecture

Intent Parsing Engine
 ├─ Language Normalizer
 ├─ Semantic Decomposer
 ├─ Ambiguity Detector
 ├─ Constraint Extractor
 ├─ Risk Classifier
 └─ Intent Validator

6. Semantic Decomposition

Each request is broken into atomic logical components: 

Verb (action)
Target (domain)
Expected Result (state change)
Unknowns (missing information)

Unknowns are preserved — never guessed.

7. Ambiguity & Missing Information Detection

  • Undefined scope
  • Multiple possible meanings
  • Missing success criteria
  • Implicit unsafe assumptions

High ambiguity blocks execution.

8. Constraint & Safety Extraction

Examples:
- "Do not change architecture"
- "Only refactor"
- "Must be backward compatible"
- "Production system"

Safety level determines whether later agents may: run commands, modify infra, or change data.

9. Risk Classification

LOW    → formatting, comments, naming
MEDIUM → logic changes, refactors
HIGH   → auth, infra, data, security

High-risk intent requires validation layers later.

10. Validation Rules

  • Objective must be measurable
  • Outcome must be verifiable
  • No internal contradictions
  • Forbidden actions are absolute

11. Output Guarantee

STEP 1 guarantees that downstream reasoning operates on intent truth, not linguistic illusion.

This is the first hallucination firewall.
