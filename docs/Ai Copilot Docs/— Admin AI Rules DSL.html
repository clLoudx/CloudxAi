Admin AI Rules DSL

Governance Language for Autonomous Software Execution

1. Purpose

Admin AI exists to prevent intelligent systems from doing stupid things at scale.

2. Core Concepts

  • Rule
  • Condition
  • Action
  • Approval
  • Audit

3. Rule Structure

RULE <rule_name>:
  IF <conditions>
  THEN <action>
  [ELSE <fallback>]

4. Conditions

environment == "production"
action == "execute"
file.path CONTAINS "/infra/"
risk_score > 0.7
user.role != "admin"

5. Actions

allow
deny
require_human_approval
log_only
rollback_last_change

6. Examples

Production Protection

RULE prod_guard:
  IF environment == "production"
  AND action == "execute"
  THEN require_human_approval

Dangerous Command Block

RULE block_rm_rf:
  IF terminal.command CONTAINS "rm -rf"
  THEN deny

Auto-Approve Safe Refactors

RULE safe_refactor:
  IF risk_score < 0.2
  AND action == "modify_code"
  THEN allow

7. Rule Evaluation Order

1. System rules
2. Organization rules
3. Project rules
4. User rules

First match wins. Explicit deny overrides allow.

8. Audit & Traceability

  • Every rule evaluation is logged
  • Every denied action is explainable
  • Every approval is traceable to a human

9. Safety Guarantees

No execution without an explicit rule path. Silence is denial.

10. Strategic Truth

This DSL is the difference between a toy AI and an enterprise-grade AI operating system.
